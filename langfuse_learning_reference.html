Langfuse Learning Notes

A practical reference for how Langfuse fits into the Travel Copilot project: sessions, traces, observations, prompts, LangChain/OpenAI integration, when to use @observe, and when not to.

Langfuse Tracing Sessions Observations Prompt Management LangChain

1. Big Picture

Session

One complete user journey. Example: planning one trip across multiple turns.

Trace

One request/turn inside that session. Example: one user message and the system response.

Observation

One step inside the trace. Example: parse input, weather lookup, currency lookup, LLM call.

Session: travel-123
├── Trace 1: "I want to travel next week for 4 days"
│   ├── parse_input
│   └── update_state
├── Trace 2: "Make it Delhi"
│   ├── parse_input
│   └── update_state
└── Trace 3: "Show itinerary under 30000 INR in EUR"
    ├── parse_input
    ├── weather_lookup
    ├── currency_lookup
    ├── fetch_prompt
    └── ChatOpenAI

2. What Langfuse Does in This Project

  • Groups all turns under one session_id.
  • Logs each user turn as a trace.
  • Logs each internal step as an observation/span.
  • Fetches the system prompt from Langfuse Prompt Management.
  • Tracks model calls, tokens, and cost via LangChain + OpenAI integration.
  • Makes debugging easier because each step can show input, output, metadata, errors, and timing.

3. Core Langfuse Functions and What They Mean

Function / Integration Purpose Typical Use Recommended?
get_client() Initialize and access the Langfuse client. Prompt fetch, current observation updates, manual observation creation. Yes
propagate_attributes(...) Attach session/user/tags/global metadata to everything inside a scope. Top-level request/turn wrapper. Yes
@observe(...) Low-noise automatic observation for a function. Business logic functions like parsing, currency lookup, weather lookup. Strongly recommended
start_as_current_observation(...) Manually create an observation/span using a context manager. Special cases requiring tight manual nesting/control. Sometimes
start_observation(...) Manual observation without with. Long-running or split lifecycle work that ends later. Rare
obs.update(...) Update the current observation object with output, metadata, level, usage. Enrich a manual observation after work completes. Yes, when doing manual tracing
update_current_observation(...) Update the active observation without passing the observation object around. Decorator-based tracing with custom metadata/output. Yes
get_prompt(...) Fetch prompt from Langfuse Prompt Management. System prompts, versioned prompts, prompt variables/config. Yes
CallbackHandler() LangChain auto-tracing for chain + model calls. LLM calls, prompt formatting, token/cost tracking. Yes

4. Reserved Fields vs Custom Fields

Langfuse has some fields with special meaning. These are not “put anywhere” fields. They should be used only where Langfuse expects them.

Reserved / special fields

FieldMeaningWhere it belongs
session_idGroups traces into one sessionpropagate_attributes(...)
user_idIdentifies the end userpropagate_attributes(...)
trace_nameName of the overall tracepropagate_attributes(...)
tagsLabels for filteringpropagate_attributes(...)
nameName of the observation@observe(...) or start_as_current_observation(...)
as_typeObservation type@observe(...) or start_as_current_observation(...)
inputInput to the observationObservation create/update
outputOutput of the observationObservation update
modelModel name for generationGeneration observation / framework integration
usage_detailsToken/cost-related usage dataGeneration update / integration
levelLog level like warning/errorObservation update

Custom fields

Your own fields should usually live inside metadata. That is your safe flexible area. 

metadata = {
  "step_type": "tool",
  "tool_name": "currency_mcp",
  "tool_status": "success",
  "from_currency": "USD",
  "to_currency": "EUR"
}
Rule: Reserved fields tell Langfuse how to understand the trace. Metadata tells you how to understand your system.

5. propagate_attributes(...)

This does not create a trace or a span. It only sets context for everything inside the block. 

with propagate_attributes(
    session_id="travel-123",
    user_id="user-1",
    tags=["travel-copilot", "openai"],
    metadata={"flow_stage": "show_itinerary", "city": "Delhi"},
):
    ...

Use it for

  • session_id
  • user_id
  • Global tags
  • High-level metadata like provider, flow stage, city

Do not use it for

  • Per-step output
  • Per-step tool payloads unless you intentionally want them on every child observation

6. start_as_current_observation(...)

This manually creates an observation (usually a span). If there is no active trace yet, the first observation starts a new trace automatically. 

with langfuse.start_as_current_observation(
    name="currency_lookup",
    as_type="span",
    input={"amount": 100, "from": "USD", "to": "EUR"},
) as obs:
    result = call_tool()
    obs.update(output=result)

Best for

  • Manual custom nesting
  • Explicit control over a root turn span
  • Cases where @observe is not enough

Why it felt noisy

Because it is the manual form. It gives control, but adds boilerplate. In most app code, @observe is cleaner.

7. start_observation(...)

This is the manual non-context-manager version. You start it now and end it later yourself. 

obs = langfuse.start_observation(name="long_running_step")
# do work across different code paths
obs.update(output={"status": "done"})
obs.end()

Use it for

  • Long-running background jobs
  • Distributed/async workflows
  • When start and end happen far apart

Do not use it for

  • Normal request-time parsing/tool/LLM steps in this project

8. obs.update(...) and update_current_observation(...)

These enrich an observation after it starts. 

# If you have the observation object
obs.update(
    output={"converted_amount": 86.82},
    metadata={"tool_name": "currency_mcp", "tool_status": "success"}
)

# If using @observe and want to update the active span
langfuse.update_current_observation(
    metadata={"tool_name": "currency_mcp"},
    output={"converted_amount": 86.82}
)

Typical things to update

  • output
  • metadata
  • level (warning/error)
  • usage_details for manual generation usage

9. @observe(...)

This is the low-noise, recommended way to instrument your own functions. It automatically creates an observation, captures inputs/outputs, timing, and errors. 

from langfuse import observe, get_client

langfuse = get_client()

@observe(name="travel.currency.lookup")
async def maybe_convert_currency(user_query: str) -> dict | None:
    langfuse.update_current_observation(
        metadata={"step_type": "tool", "tool_name": "currency_mcp"}
    )
    return {"amount": 100, "from_currency": "USD", "to_currency": "EUR", "converted_amount": 86.82}

Values you can pass to @observe

  • name — always set this in practice
  • as_type — usually span; sometimes generation

Best practice

  • Always use semantic names like travel.weather.lookup, not raw function names like process.
  • Add custom metadata inside the function using update_current_observation(...).

11. Prompt Integration

Langfuse is not only for tracing. It is also your prompt registry. 

client = get_client()
prompt = client.get_prompt("travel_copilot_system")
compiled_prompt = prompt.compile(
    session_state="...",
    user_request="...",
    tool_context="...",
    guardrails="..."
)

Why this matters

  • Prompt text is not hardcoded.
  • Prompt versions can be changed in Langfuse UI.
  • Prompt usage can be associated with traces/generations.

12. LangChain / OpenAI Integration

For LLM calls, the best practice is to use the native Langfuse LangChain integration instead of manually tracing model calls. 

from langfuse.langchain import CallbackHandler

handler = CallbackHandler()
response = chain.invoke(
    prompt_variables,
    config={
        "callbacks": [handler],
        "run_name": "travel_itinerary_generation"
    }
)

This gives you

  • PromptTemplate step
  • ChatOpenAI step
  • Token usage
  • Cost
  • Chain nesting
Do not manually create a separate generation observation for the same LangChain/OpenAI call, or the trace can become confusing and look like duplicated cost.

13. Why Cost Looked “Double”

Langfuse often shows cost at multiple levels of the hierarchy:

  • Leaf level: actual OpenAI call cost
  • Parent level: rolled-up total from child observations

So the parent cost is not an extra bill. It is an aggregate.

14. Best-Practice Integration Pattern for This App

with propagate_attributes(
    session_id=state.session_id,
    user_id=state.user_id,
    tags=["travel-copilot", provider],
    metadata={"flow_stage": state.flow_stage},
):
    # decorated business logic functions
    parse_input(...)
    maybe_get_weather(...)
    maybe_convert_currency(...)
    get_prompt(...)

    # LangChain + CallbackHandler for LLM
    chain.invoke(...)

Recommended split

  • propagate_attributes → top-level session/user/global context
  • @observe → your business logic
  • update_current_observation → extra metadata/output inside observed functions
  • CallbackHandler → model/framework tracing
  • get_prompt → prompt management

15. What Not to Overuse

  • Do not wrap every small line with manual with start_as_current_observation(...).
  • Do not put all custom keys at the top level; put them in metadata.
  • Do not manually create extra LLM generation spans if LangChain callback already traces the model call.
  • Do not use start_observation(...) for normal request-time business logic.

Langfuse Quick Cheat Sheet

Minimal Pattern

with propagate_attributes(session_id="..."):
    answer_user(...)

@observe Equivalent (Important)

@observe internally does:

with start_as_current_observation(name="...") as obs:
    obs.update(input=function_args)
    result = function(...)
    obs.update(output=result)

How to Add More Values in @observe

You enrich the same observation using:

from langfuse import get_client
langfuse = get_client()

@observe(name="travel.currency.lookup")
def func():
    langfuse.update_current_observation(
        metadata={
            "tool_name": "currency_mcp",
            "status": "success"
        },
        output={"extra": "data"}
    )

Key Rules

  • Use name always
  • Use metadata for custom fields
  • Use CallbackHandler for LLM
  • Use propagate_attributes once per request

Golden Mental Model

Session → propagate_attributes
Trace → first observe/span
Step → @observe
Details → update_current_observation
